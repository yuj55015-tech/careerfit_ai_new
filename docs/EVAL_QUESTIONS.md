## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.



## 프로젝트 전체

- 이 서비스가 해결하는 문제는 무엇인가?
    취업을 준비하는 대학생이 느끼는 취업에 대한 막막함을 해결하고자 한다. 특히 관심 있는 직무에 어떤 역량이 필요한지 몰라서 막막해하는 경우가 많은데, CareerFit AI는 전공·보유 스킬·관심 직무를 입력하면 실제 채용 공고와 공모전 데이터를 근거로 지금 어떤 역량이 부족한지, 어떤 방향으로 준비하면 좋을지를 구체적으로 알려준다. 그냥 "이런 걸 준비하세요"라는 뻔한 조언이 아니라, 실제 데이터를 근거로 답해준다는 점에서 막연한 불안감을 좀 더 실질적인 방향 제시로 바꿔주는 게 이 서비스의 목표다.
- RAG를 사용하지 않으면 어떤 문제가 생기는가?
    최신 정보를 학습하지 못해 잘못된 정보를 줄 수 있다. LLM은 학습 시점 이후의 정보(최근 채용 공고나 공모전 마감일 같은 것)를 알지 못하기 때문에, RAG 없이 LLM에게 바로 물어보면 실제로는 존재하지 않는 회사나 공고를 지어내거나, 이미 지난 공모전을 추천하는 등 부정확한 답변을 할 위험이 있다. RAG를 쓰면 실제로 가지고 있는 공고 데이터를 먼저 검색해서 그걸 근거로 답변을 만들기 때문에, 이런 문제를 줄이고 답변에 실제 출처(sources)를 붙일 수 있게 된다.
- 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?
    기술스택의 역할에 대한 이해가 없어서 이해하는데 오래 걸렸다. FastAPI, Gemini API, Pandas, SQLite, ChromaDB처럼 각자 다른 역할을 하는 기술들이 하나의 파이프라인 안에서 어떻게 이어지는지 처음엔 감이 잘 안 왔다. 특히 SQLite와 ChromaDB를 왜 둘 다 써야 하는지, RAG라는 게 정확히 어느 단계에서 어떤 역할을 하는지가 헷갈렸다. 코드를 하나씩 뜯어보고 각 서비스 파일(rag_service.py, llm_service.py)이 실제로 어떤 입출력을 주고받는지 직접 따라가면서, 그리고 관련 문서와 자료를 찾아보면서 각 기술의 역할을 하나씩 정리해나가며 이해했다.

## 데이터 관련

- 왜 이 데이터를 선택했는가?
    CareerFit AI의 목적이 "전공·스킬·관심 직무 → 맞춤 커리어 조언"이라서, 데이터 분석/AI·ML/백엔드/마케팅/공공 등 다양한 직무군을 커버하는 채용공고·공모전 데이터가 필요했습니다.
- 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?
    jobs.csv를 보면 회사명이 "테크스타트업A", "금융서비스B", "AI솔루션C"처럼 익명화·가상의 이름입니다. 즉 실제 채용 사이트를 크롤링한 데이터가 아니라 학습용으로 직접 만든 샘플/목업 데이터입니다.
- SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?
    SQLite: 구조화된 정확 매칭/집계용. 예: job_type별 개수 세기, required_skills LIKE '%Python%' 같은 조건 필터링(preprocess.py의 query_sqlite()에서 실제로 이렇게 검증).
    ChromaDB: 자연어 임베딩 기반 의미 검색용. 사용자가 "통계학과 학생에게 적합한 공고"처럼 정확한 키워드가 아닌 자연어로 질문해도, 의미상 가까운 문서를 벡터 거리로 찾아줍니다.
- 결측치와 중복을 어떻게 처리했는가?
    결측치: handle_missing()에서 핵심 컬럼인 title, required_skills가 비어있는 행은 통째로 제거(이게 없으면 RAG 검색 자체가 의미 없기 때문). 나머지 텍스트 컬럼(preferred_skills, description, company, job_type)은 빈 문자열로 채움.
    중복: remove_duplicates()에서 company + title 조합이 같으면 중복으로 판단하고, 첫 번째 행만 남기고 제거(drop_duplicates(keep="first")). 실제로 jobs.csv에 "AI솔루션C - 머신러닝 엔지니어"가 중복 등록되어 있어서 이 로직이 실제로 걸러줍니다.


## RAG 관련

- RAG 구조에서 ChromaDB는 어떤 역할을 하는가?
    RAG의 "R"(Retrieval) 부분을 담당합니다. rag_service.py의 search_documents()가 사용자 질문을 임베딩하고, ChromaDB에 저장된 공고 문서들과의 벡터 거리를 계산해 가장 유사한 문서 n개를 반환합니다. 이 문서들이 이후 LLM 프롬프트의 "근거 자료"가 됩니다.
- n_results 값을 높이면 어떤 트레이드오프가 생기는가?
    높이면: LLM이 참고할 정보가 많아져 더 다양한 추천이 가능하지만, 프롬프트가 길어져 응답 속도·비용이 늘고, 유사도가 낮은(관련성 떨어지는) 문서까지 섞여 들어가 답변이 산만해지거나 부정확해질 위험이 있음 
    낮추면: 빠르고 저렴하지만, 정말 관련 있는 공고를 놓칠 수 있음
    현재 기본값(n_results=3)은 데이터셋이 15개 안팎으로 작기 때문에 적절한 균형점으로 설정된 값입니다.
- 검색 결과 품질을 어떻게 검증했는가?
    data/test_search.py에서 "데이터 분석 직무에 Python이 필요한 공고", "통계학과 학생에게 적합한 취업 공고", "백엔드 개발자 채용 공고" 같은 테스트 질문을 만들어 실제로 어떤 공고가 검색되는지, 그리고 distance(유사도 거리, 낮을수록 유사) 값을 직접 눈으로 확인하며 검증했습니다. 정량 지표(Ragas 등)까지는 아직 없고, 정성적으로 "질문 의도와 결과가 맞는지" 눈으로 확인한 수준입니다. (README의 "향후 개선"에 Ragas 도입 계획을 넣어둔 이유이기도 합니다.)    
- Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?
    Hallucination은 LLM이 실제로는 근거가 없는 정보(존재하지 않는 회사, 조건, 통계 등)를 그럴듯하게 지어내는 현상입니다. RAG는 답변을 생성하기 전에 실제 문서(공고 데이터)를 먼저 검색해서 그 내용을 프롬프트에 근거로 포함시키고, llm_service.py의 build_rag_prompt()에서 "검색된 데이터에 없는 회사명, 조건, 공모전 정보는 지어내지 마세요", "참고 데이터가 부족하면 '제공된 자료만으로는 판단하기 어렵습니다'라고 말하세요"처럼 명시적으로 지시함으로써, LLM이 근거 없는 내용을 만들어내는 것을 줄입니다.
## API 관련

-/analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?

    사용자의 전공·스킬 목록·관심 직무 같은 입력 데이터(body) 를 서버로 보내야 하기 때문입니다. GET은 URL 쿼리 파라미터로 데이터를 전달해 리스트 형태(skills: List[str])나 긴 텍스트를 다루기 불편하고, 또한 이 요청은 매번 LLM 호출이라는 부수효과(side effect) 를 일으키는 작업(조회가 아니라 "분석을 수행"하는 행위)이라 의미상으로도 POST가 적절합니다.

-응답 JSON에 sources를 포함시킨 이유는 무엇인가?

    AI 답변의 신뢰도를 높이고 근거를 투명하게 보여주기 위해서입니다. build_sources()에서 어떤 회사·직무·필요 스킬을 근거로 답변했는지 배열로 반환하는데, 이렇게 하면 사용자가 "AI가 왜 이렇게 조언했는지"를 직접 확인할 수 있고, RAG 특유의 강점(근거 기반 답변)을 사용자에게 명시적으로 드러낼 수 있습니다.

-/health 엔드포인트가 왜 필요한가?

    서버가 정상적으로 살아있는지 확인하기 위한 용도입니다. Docker 컨테이너의 헬스체크, Render 같은 배포 플랫폼의 모니터링, 또는 프론트엔드가 백엔드 연결 상태를 미리 확인하는 용도로 씁니다. 실제 비즈니스 로직 없이 {"status": "ok"}만 반환하는 것도 그 때문(가볍고 빠르게 응답해야 함).


## 보안 관련

-.env 파일을 GitHub에 올리면 어떤 일이 생기는가?

    GEMINI_API_KEY, MISTRAL_API_KEY 같은 실제 API 키가 공개 저장소에 노출됩니다. 노출되면 제3자가 그 키로 API를 무단 호출해 요금이 청구되거나, 키가 정지·악용될 수 있습니다. 그래서 .gitignore에 .env를 등록해 커밋 자체가 안 되도록 막는 게 기본입니다.

-.env와 .env.example의 차이는 무엇인가?

    .env: 실제 API 키 값이 들어있는 파일. Git에 절대 올라가면 안 됨 (.gitignore에 포함).
    .env.example: 실제 키값 없이 어떤 환경변수가 필요한지 형식만 보여주는 템플릿 파일(GEMINI_API_KEY=your_gemini_api_key_here). 이건 GitHub에 올려도 안전하고, 다른 개발자가 프로젝트를 클론했을 때 "아, 이런 값들을 채워야 하는구나"를 알 수 있게 해주는 가이드 역할을 합니다.


## Docker 관련

-Docker를 사용하는 이유는 무엇인가?

    "내 컴퓨터에선 되는데 다른 곳에선 안 돼요" 문제를 없애기 위해서입니다. Python 버전, 패키지 버전, OS 차이와 상관없이 Dockerfile에 정의된 환경 그대로 어디서든(로컬, Render, 다른 서버) 동일하게 실행되도록 컨테이너로 패키징합니다. 배포 플랫폼(Render 등)에 올릴 때도 "이 이미지 그대로 실행해줘"라고 넘기기만 하면 되니 배포가 단순해집니다.

-Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?

    EXPOSE 8000: 컨테이너 내부에서 이 앱이 8000번 포트를 사용한다는 걸 문서적으로 선언하는 것. 실제로 포트를 여는 것은 아니고(실행 시 -p 8000:8000처럼 외부와 매핑해야 접속 가능), 컨테이너를 사용하는 사람/도구에게 "이 포트를 쓸 거야"라고 알려주는 역할입니다.
    CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]: 컨테이너가 시작될 때 자동으로 실행할 명령어. --host 0.0.0.0으로 지정해야 컨테이너 밖(외부)에서 접속 가능하고(127.0.0.1로 하면 컨테이너 내부에서만 접속 가능), 이 CMD 덕분에 docker run만 하면 별도 명령 없이 서버가 바로 뜹니다.
